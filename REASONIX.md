# REASONIX.md — Reasonix Code 项目协作规范

> 本文件定义 Reasonix Code（以下简称 **Reasonix**）在本项目中的工作方式。
> 与 `docs/协作规范.md`（项目业务约定）互补，本文件侧重 **Reasonix 工具使用纪律** 与 **协作流程**。
> 用户说「按 REASONIX.md 来」时，直接遵守以下全部条款。

---

## 一、设计先行（最高优先级）

适配 Reasonix 工具集的设计先行流程：

1. **任何代码动作前，先用 Reasonix 机制出设计、等认可**。顺序固定：
   `读上下文 → 出设计 → 用户认可 → 实现 → 回填文档`。

2. **用 `submit_plan` 提交设计**。Plan 正文写清：
   - 改动的**每个文件**及改动内容
   - 创建的**每个新类/函数**的职责
   - 设计思路与取舍理由
   - **Open questions**（不确定的命名、方案取舍等，让用户填空决策）

3. **用 `ask_choice` 处理分支决策**。当方案有 2+ 合理选型时：
   - 先分析各方案优劣
   - 用 `ask_choice` 让用户选，不在 prose 里列菜单
   - 用户选定后，用选定方案走 `submit_plan`

4. **用 `todo_write` 跟踪多步骤实现进度**。当任务拆成 3+ 步骤时：
   - 每步标记 `in_progress` → 完成后改 `completed`
   - 跨步骤切换时先完成当前步再改状态

5. **产出的设计文档写回 `docs/详细设计/`**，操作记录写回 `docs/操作记录/`。

6. **未经认可，不擅自 edit_file / write_file / run_command（只读除外）**。

## 二、流程纪律

7. **与项目现有规范保持一致**：本项目已有的约定见 `docs/协作规范.md`、`ARCHITECTURE.md`、
   `knowledge/*.md`。Reasonix 不与之冲突，如有疑义先问用户。

8. **不越界**：用户说「只做 X」就只做 X。不提前做后续步骤，不顺手修无关问题。
   发现非改不可的问题 → 在 Open questions 中提出，等用户决定。

9. **作废即停**：用户说「作废 / 停 / 算了」立即停止当前所有动作，不续做、不补刀。

10. **决策固化**：用户做过的选择（命名、目录、方案取舍）用 `remember(scope:project)`
    记下来，后续不再反复问。

11. **不臆测关键参数**：连接串、密码、路径等不猜测、不内置默认值；缺失时显式报错或
    问用户。

## 三、Reasonix 工具使用约定

### submit_plan（设计审批门禁）

| 场景 | 必须 submit_plan | 直接做 |
|------|:---:|:---:|
| 多文件重构、架构变更 | ✅ | — |
| 涉及新增/删除文件 | ✅ | — |
| 破坏性改动（迁移、删除代码） | ✅ | — |
| 用户需求模糊，有多个解读 | ✅ | — |
| 一行 typo 修复 | — | ✅ |
| 明显 bug 有明确修复方案 | — | ✅ |
| 纯只读探查（search/read/grep） | — | ✅ |

### ask_choice（分支决策）

- 方案有 2~4 个合理方向 → 用 `ask_choice`，不用 prose 列菜单
- 允许 `allowCustom:true` 防止选项没覆盖用户想法
- 调用后停止，等用户选了再继续

### todo_write（进度跟踪）

- 任务拆成 3+ 步骤时必用（记录步骤、显式切换 in_progress/completed）
- 不替代 `submit_plan`（plan 是给用户看的审批，todo 是自己的进度条）

### remember（持久化决策）

- **scope:project** → 项目级决策、命名约定、用户偏好
- **priority:high** → 铁律级约束（如「未经 submit_plan 不写代码」）
- 不要记 transient 任务状态（那是 todo_write 的事）

### 编辑规范

- SEARCH/REPLACE 块必须 `read_file` 后精确匹配，不猜测文件内容
- 新文件用 `write_file`，已有文件用 `edit_file` 或 `multi_edit`
- 同一文件多处改动优先用 `multi_edit`（原子提交，一处失败全部回滚）

## 四、代码质量标准

12. **单文件 ≤ 200 逻辑行**（不含 import/文档注释/空行/纯括号行），目标 100 行左右。
    超出按职责拆分包/模块。

13. **每个文件顶部写模块 docstring**：用途 + 对应设计文档 + 用法示例（中文）。

14. **每个类/函数写清做什么 + 设计思路**（中文注释，与现有风格一致）。

15. **实现前先读 `knowledge/*.md`** 了解已有翻译模式，不重复造轮子。
    确需新方案时在设计文档说明原因。

16. **写完即自检**：跑相关测试/脚本验证通过情况，逐项向用户汇报。
    - 叶子翻译改动 → `python scripts/check.py leaf`
    - ASG 改动 → `python scripts/check.py asg`
    - 全量 → `python -m pytest tests/ -v && python scripts/check.py all`

17. **输出产物落盘到 `output/` 下子目录**，不用临时文件，不清空已有基线。

## 五、Context 经济

18. **大文件精读优先**：超过自动预览阈值的文件先用 `head:N` / `range:"A-B"` / `search_content`
    定位目标区域，不无脑全文读。

19. **善用 `explore` subagent**：跨 5+ 文件的调查或只需要结论的调查，用
    `run_skill({ name: "explore", arguments: "..." })` 隔离到子会话。

20. **任务收尾做 Token 分析**：在该任务操作记录末尾追加「Token 使用分析」节，
    记录主要消耗来源与量级。

21. **Context 偏高时主动预警**：在回复正文提醒用户建议 `/clear` 或新开会话。
