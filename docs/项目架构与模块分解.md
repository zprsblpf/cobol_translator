# 项目架构与模块分解

> 本文档是对项目的完整架构分析，包含功能模块划分、数据流、类职责和依赖关系。
> 相对于 ARCHITECTURE.md，本文更侧重**模块间数据流**和**每条翻译路径的实际执行流程**。

---

## 一、总览：三相三路径架构

```
COBOL .cob  →  相1: 预处理层  →  相1.5: 解析层  →  相2: ASG层  →  相3: 翻译层  →  Java
```

**三条翻译路径（当前并行运行，逐步收敛）：**

| 路径 | 骨架层 | 叶子层 | 当前状态 |
|------|--------|--------|---------|
| 🅰 旧路 (rules.py) | `rules.build_section` + `segmenter` | `translator.leaf.*` | ✅ 完整，主线在用 |
| 🅱 ASG Visitor 新路 | `asg.section_visitor.SectionJavaVisitor` | `translator.leaf.*`（共用） | ✅ 完整，旁路比对 |
| 🅲 skeleton_gen 骨架路 | `skeleton_gen.render_skeleton` | 仅骨架（TODO），无叶子填充 | ✅ 用于 no-llm 模式 |

## 二、功能模块分解

### 模块 1：预处理层（preprocess/）

**职责**：原始 COBOL 行级清洗，产出净化后的代码行流。

| 文件 | 类/函数 | 职责 | 输入 | 输出 |
|------|---------|------|------|------|
| `line_stream.py` | `CleanSource` | 行级净化：去序号列、去停用行(!!!!!!)、去注释行(*)、续行合并 | raw lines | (lineno, clean_line) pairs |
| `columns.py` | (工具函数) | 定列格式判定：is_comment/is_deactivated/Area A/B | raw line | bool |
| `dialect.py` | (归一化函数) | 方言归一：GO→GO TO（对引号内文本免疫） | code lines | normalized lines |
| `line_stream.py` | `build()` | 完整预处理入口：strip → normalize → 续行合并 | raw lines | CleanSource |

**测试覆盖**：`test_translation.TestDialectNormalize`（5 tests）

### 模块 2：解析层（parser/）

**职责**：把清洗后的 COBOL 解析为结构化数据类型（无 LLM，纯确定性）。

| 文件 | 类/函数 | 职责 | 输入 | 输出 |
|------|---------|------|------|------|
| `cobol_parser.py` | `parse()` | 主入口：定位 DIVISION → 解析WS变量 → 解析SECTION → 收集COPY | .cob path | `CobolProgram` |
| | `_parse_variables()` | 合并续行 → 逐行解析 Variable（level/name/pic/comp/occurs/redefines） | WS/LINKAGE 行 | `Variable[]` |
| | `_parse_sections()` | SECTION 切分 + 提取 PERFORM/CALL/GO TO | 过程体行 | `CobolSection[]` |
| | `_strip_cobol_line()` | 委托 line_stream.CleanSource.strip_line | raw line | clean line |
| `variable_resolver.py` | `resolve()` | Variable[] → JavaField[]（去重保首现） | `Variable[]` | `JavaField[]` |
| | `_get_java_type()` | PIC/COMP → Java 类型（委托 spec_loader） | pic, comp | type string |
| | `generate_field_declarations()` | 生成 Java 字段声明代码块 | `JavaField[]` | code string |
| | `generate_grouped_field_declarations()` | 按 01 组分块的字段声明（State 类用） | `Variable[]` | code string |
| | `build_field_type_map()` | {java名 → {type, is_array, array_size}} | `JavaField[]` | dict |
| | `generate_variable_context()` | 变量名映射摘要（供 LLM 提示） | `JavaField[]` | string |
| `pipeline.py` | `build_parse_result()` | 一键解析 + 变量解析 + 组装 state 增量 | .cob path | state dict |

**测试覆盖**：`test_translation.TestVariableResolver`（20 tests）
**依赖**：`config.spec_loader`（类型映射）、`config.conversions`（命名转换）、`preprocess.line_stream`（行清洗）

### 模块 3：ASG 构建层（asg/）

**职责**：把解析产物的裸语句(Stmt)提升为带类型的 AST 节点树。

| 文件 | 类/函数 | 职责 | 输入 | 输出 |
|------|---------|------|------|------|
| `nodes.py` | *(所有 dataclass)* | 类型节点定义：Program/Section/Paragraph + 语句节点(MoveStmt/CallStmt/GotoStmt/IfStmt/PerformStmt/EvaluateStmt/...) + IO 吸收节点(BegnForeachStmt/BegnSingleStmt/IoReadSingleStmt/IoWriteSingleStmt) | — | 节点类型 |
| `registry.py` | `ProcRegistry` | 过程注册表：resolve() / resolve_thru() 按名/区间解析 PERFORM 目标 | proc_order 四元组 | ProcRef/ProcUnit[] |
| | `SymbolTable` | 符号表：变量名查找（暂未广泛使用） | Variable[] | lookup() |
| | `build()` | 一次性注册表构建（相2入口） | CobolProgram | (ProcRegistry, SymbolTable) |
| `builder.py` | `build()` | 主入口：split_paragraphs → segment → _lift → Program 节点树 | CobolProgram | Program |
| | `_lift()` | Stmt 调度器：按 tokens[0] 分派到 MoveStmt/CallStmt/GotoStmt/... | Stmt | 类型节点 |
| | `build_paragraphs()` | 从原始 paragraph 构建 Paragraph 节点（skeleton_gen 入口用） | paras_raw + ctx | Paragraph[] |
| `visitor.py` | `LeafJavaVisitor` | 叶子 visitor 基类：visit_MoveStmt/visit_CallStmt/visit_IfStmt/visit_EvaluateStmt/... | ASG 节点 | Java lines |
| `section_visitor.py` | `SectionJavaVisitor` | 骨架装配 visitor：render_section/render_paragraphs/状态机/PERFORM目标解析/IO吸收渲染 | Section + ctx | Java lines |
| `structure_rewrite.py` | `rewrite_structures()` | 结构吸收总入口：BEGN+NEXTR→foreach / 单次BEGN / READR / WRITE | Paragraph[] | Paragraph[] |
| | `rewrite_begn_foreach()` | BEGN+NEXTR 自跳循环识别 → BegnForeachStmt | Paragraph[] | Paragraph[] |
| | `rewrite_begn_single()` | 单次 BEGN 等值定位 → BegnSingleStmt | Paragraph[] | Paragraph[] |
| | `rewrite_readr_single()` | READR/READS 单条读 → IoReadSingleStmt | Paragraph[] | Paragraph[] |
| | `rewrite_write_single()` | WRITR/UPDAT/DELET → IoWriteSingleStmt | Paragraph[] | Paragraph[] |

**测试覆盖**：ASG 相关测试约 60+ tests（含 diff 比对、IO 吸收、visitor 渲染、section visitor）
**设计文档对应**：步骤17-旁路建相2-ASG、步骤18~24C-绞杀项3/4、步骤25~28 结构吸收

## 三、数据流（完整翻译流程）

```
cob 文件
  │
  ▼
[parse_cobol_node]
  ├── parser.cobol_parser.parse() → CobolProgram
  ├── parser.variable_resolver.resolve() → JavaField[]
  ├── parser.variable_resolver.build_field_type_map() → field_type_map
  ├── parser.variable_resolver.generate_field_declarations() → code
  └── → state: {sections_meta, field_type_map, java_field_names, ...}
  │
  ▼
[build_context_and_skeleton_node]
  ├── spec_loader.io_programs() / date_programs() / system_programs()
  ├── spec_loader.io_default_pattern()
  ├── analyzer.callgraph.build_call_graph() → call_graph
  ├── analyzer.dataflow.analyze_dataflow() → var_lifecycle
  ├── translator.skeleton._assign_sections_to_modules() → modules
  ├── translator.skeleton._build_state_class() → State.java
  ├── translator.skeleton._build_module_skeleton() → ModuleXxx.java
  ├── translator.skeleton._build_facade_skeleton() → Service.java
  ├── translator.naming.build_struct_registry() → struct_registry
  └── → state: {modules, call_graph, state_class_source, ...}
  │
  ▼
[translate_section_node]  ← 对每个 SECTION 并行执行
  ├── translator.segmenter.segment() → Stmt[]
  ├── translator.segmenter.split_paragraphs() → [(label, lines)]
  │
  ├── [主路径: ASG]  ← 当前默认（fallback 到旧路）
  │   ├── asg.builder.build_paragraphs() → Paragraph[]
  │   ├── asg.structure_rewrite.rewrite_structures() → Paragraph[]
  │   └── asg.section_visitor.SectionJavaVisitor.render_paragraphs() → Java
  │
  ├── [备路径: 旧路]  ← ASG 失败时
  │   ├── translator.skel.io_rewrite.rewrite_io_paras() → paras
  │   ├── translator.rules.build_section() → Java（含骨架+叶子）
  │   └── translator.leaf.* → translate_*() → Java
  │
  └── → state: {translated_sections: {method_name: body}}
  │
  ▼
[assemble_node]
  ├── assemble.assemble_outputs()
  │   ├── _fill_stubs() → 替换 TODO 占位
  │   ├── 写出 State.java / Service.java / Module*.java
  │   └── → call_graph.md + review_checklist.md
  └── → Java 文件 + 审查清单
```

## 五、关键缺口与风险点

### 缺口 1：入口控制流（execute() 方法未自动拼接）

当前 `XxxService.java` 的 `execute()` 方法内容是 TODO：
```java
public void execute() {
    XxxState wsaa = new XxxState();
    // TODO 入口控制流待译：...
}
```

`call_graph` 已经计算了 `entry_sequence`（入口段序列），但还没有代码自动将其转为 execute() 的方法体。

**位置**：`translator/skeleton.py` → `_build_facade_skeleton()`

### 缺口 2：GO TO DEPENDING ON 未实现

`docs/翻译标准/流程结构.md` 中明确定义了 GO TO DEPENDING ON 的保守策略（TODO 占位），但规则引擎里仍是 `// TODO-GOTO-DEPENDING`。

**位置**：`translator/leaf/control.py`

### 缺口 3：88 条件名解析不够完整

当 88 条件名的 holder 字段在 WS 树中可以解析到时能翻译为等价条件，但复杂嵌套场景（跨 COPY 引用）可能解析失败。

**位置**：`translator/leaf/cond.py`

### 缺口 4：SEARCH 动词未完成

SEARCH ALL / SEARCH VARYING 当前全 fallthrough 到 LLM，无确定性规则。

**位置**：`translator/leaf/search.py`

### 缺口 5：逻辑文档生成（你提出的核心需求）

这是完全缺失的功能——从 ASG 分析程序逻辑路径，生成可理解的"处理逻辑树"文档。

### 风险点 1：131 个 SECTION 的大程序翻译质量

ZPOLDWNM.cob 有 131 个 SECTION、886 个 WS 变量、134 个 COPY 引用——这是真实的保险行业程序规模。当前测试只覆盖了 minimal.cob（2 个 SECTION）和合成片段。

### 风险点 2：两条路径一致性
