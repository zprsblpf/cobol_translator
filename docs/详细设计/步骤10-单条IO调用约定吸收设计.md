## 0. 已确认的决定 / 待确认决策点

### 已确认（来自用户下发的翻译标准）
- `xxx-PARAMS` = **该 PO（实体）的全部字段**，不是凭空的"参数包"。CALL 前对 `xxx-<字段>` 的 MOVE 是在给 PO 字段赋值 / 构造查询键。
- `MOVE SPACES TO xxx-PARAMS`（组级）→ 该 PO **所有字段 = `""`**（**不是** `new`，因 Java new 对象字段默认 `null` ≠ 空串）。
- `INITIALIZE xxx-PARAMS` → 该 PO **按字段类型初始化**（数值→`0`/`BigDecimal.ZERO`，字符→`""`）。
- `xxx-FUNCTION` / `xxx-FORMAT` / `xxx-STATUZ` **不是真实字段**，是调用约定，须**吸收**（不生成 setter / getter）。
- `READR` → 按键随机读**一条**：`Record r = repo.findBy<真实键>Readr(键实参…)`；`STATUZ = O-K` ⇔ `r != null`。
- STATUZ 状态码：`O-K`=查到数据（ok）；`MRNF`/`ENDP`=没有数据（not ok）。

### ✅ 决策定案（2026-06-04 用户拍板）
- **决策 A（STATUZ 真异常分支）= 异常机制 + 调 580**：READR 拆三态——`record != null`=O-K（查到）；`record == null`=MRNF（没查到，**正常分支**）；repository **抛异常**=真 DB 错误 → `catch` 中调 `dbError580()`（对应原 `PERFORM 580-DB-ERROR`）。
- **决策 B（键字段来源）= B1**：`findBy<键>Readr` 的键字段 = CALL 前那几行 `MOVE … TO xxx-<字段>` 的目标字段，**按出现顺序**拼。
- **决策 C（组级 params 互拷）= 同名字段互拷**：`MOVE xxx-PARAMS TO yyy-PARAMS` → 对两个 PO 的**同名字段**逐个 `yyy.setF(xxx.getF())`；只拷两边都有的同名字段，名字对不上的忽略。
- **决策 D（运行时分发退路）= 移除**：功能码**一定在 CALL 前显性字面量赋值**（READR/BEGN/NEXTR/UPDAT…），不存在"藏在 WS 变量、运行时才知道"的情况 → 静态恒可得。故 `rules.py:1027` 的 `execute(obj.getFunction(), obj)` 兜底分发是**为不存在场景写的死代码，移除**。

## 2. 问题诊断（根因，来自 1500-READ-ELPO 实测）

`translator/rules.py:1000` 的 `_t_call` docstring 写明当前 READR 契约：

> 契约：参数对象进出（Repository 在返回对象上写 STATUZ），下游 STATUZ 检查零改写。

即 READR **故意不吸收**——把整套调用约定原样搬进 Java（`obj = repo.method(obj)`，`rules.py:1024`
把 `{key}` 直接替换成整个 `elpoParams`）。`_t_move`（`rules.py:1066`）也注释"仍照常生成 setFunction/setFormat setter"。

**三者不一致**：knowledge 标准（`io_call_patterns.md:54` 已写 READR→record!=null）✅、BEGN 实现 ✅、**READR 实现 ❌**。

`1500-READ-ELPO` 译文的 6 项违规（同一根因的派生）：

| # | COBOL | 当前译文 | 应为 |
|---|---|---|---|
| 1 | `MOVE SPACES TO ELPO-PARAMS` | `elpoParams = new ElpoParams();` | PO 所有字段 `""` |
| 2 | `MOVE READR TO ELPO-FUNCTION` | `setFunction("READR")` | 吸收（→ 方法名后缀 `Readr`） |
| 3 | `MOVE ELPOREC TO ELPO-FORMAT` | `setFormat(st.elporec)` | 吸收 |
| 4 | `CALL 'ELPOIO' USING ELPO-PARAMS` | `elpoParams = repo.findByKeyReadr(elpoParams)` | `ElpoRecord elpo = repo.findByChdrcoyAndChdrnumReadr(…)` |
| 5 | `IF ELPO-STATUZ NOT=O-K …` | `!"O-K".equals(elpoParams.getStatuz())` | `elpo == null` |
| 6 | `MOVE ELPO-PARAMS TO SYSR-PARAMS` | `sysrParams = elpoParams;`（编译不过） | 见决策 C |

## 3. 设计方案（确定性规则，无 LLM）

### 3.1 MOVE / INITIALIZE 对 PARAMS 的处理（改 `_t_move` + 新增 INITIALIZE 处理）
- `MOVE SPACES TO xxx-PARAMS`（组级，目标是 PARAMS 结构）→ 该 PO 所有字段置 `""`（按字段类型：字符 `""`、数值 `0`）。
- `INITIALIZE xxx-PARAMS` → 同上，按字段类型初始化。
- `MOVE 字面/字段 TO xxx-FUNCTION` → **吸收**：仅记入 `ctx.struct_function[前缀]`（已有机制），**不生成 setter**（去掉现有"照常生成 setter"）。
- `MOVE xxxREC TO xxx-FORMAT` → **吸收**：不生成 setter（FORMAT 仅用于校验记录格式，Java 侧无对应）。
- `MOVE 值 TO xxx-<真实键/数据字段>` → 暂存为"查询键备料"（供 3.2 拼 finder 实参），不立即生成 setter。

### 3.2 CALL 单条操作（改 `_t_call` READR 分支，`rules.py:989`）
- 由 `ctx.struct_function[prefix]` 取功能码；属单条操作集合（READR/UPDAT/…）→ 走**吸收路径**。
- 生成：`<RecordType> <var> = <repo>.findBy<键字段序列>Readr(<键实参>);`
  - `RecordType` / `var`：见决策 B / 命名（PO 类型）。
  - `键字段序列` + `键实参`：由 3.1 暂存的"查询键备料"按出现顺序拼（决策 B1）或读 io_mappings（决策 B2）。
- 不再 `obj = repo.method(obj)`，不再把整个 params 当 key。
- 删除 `rules.py:1027` 的 `execute(obj.getFunction(), obj)` 运行时分发退路（决策 D：功能码恒为显性字面量，该退路为死代码）。

### 3.3 STATUZ 检查改写（新增 condition 改写，挂 `_try_condition` / IO 上下文）
- 识别"紧跟 CALL 的 `IF xxx-STATUZ …`"，按 PO 变量改写：
  - `xxx-STATUZ = O-K` → `<var> != null`
  - `xxx-STATUZ NOT = O-K` → `<var> == null`
  - `xxx-STATUZ = MRNF`（或 `= ENDP`）→ `<var> == null`
  - 真异常组合 `NOT = O-K AND NOT = MRNF`（决策 A）→ 不再生成 IF 判断；改由 `try { CALL+后续 } catch (Exception e) { dbError580(); }` 承接：CALL 整段包进 try，原 `PERFORM 580-DB-ERROR` 移入 catch。

### 3.4 组级 params 互拷（决策 C 落地后实现）
- `MOVE xxx-PARAMS TO yyy-PARAMS` → 按决策 C 译（不可退化成引用别名）。

## 4. 落地清单（已实现，回填）

实现架构定案：**结构化吸收**（镜像已有的 BEGN 单次 `_match_begn_single`/`_render_begn_single`），
在段内重写层 `_rewrite_begn_loops` 的 Pass 2 处理「setup + CALL + IF STATUZ」整段。
理由：① 决策 A 的 try/catch 是跨语句结构改写，叶子级 `_t_call`/`_t_move` 做不到；
② 与 BEGN 路径同构，复用 `_stmt_call_io`/`_stmt_touches_pfx`/`_tag_rebind`/`build_skeleton`；
③ finder 命名与实参在代码里按键派生（同 BEGN，不读 `io_default_pattern` 通用占位模板名），

### 实现细节与边界
- **决策 A（error 形）**：`IF pfx-STATUZ NOT=O-K AND pfx-STATUZ NOT=MRNF/ENDP` + then(`PERFORM 580`) →
  `try { Rec r = repo.findBy…Readr(…); <IF 之后的后续语句> } catch (Exception e) { dbError580(); }`；
  consume_to=段末（后续并入 try）。
- **决策 B1（键）**：CALL 前所有 `MOVE val TO pfx-<字段>`（排除 FUNCTION/FORMAT/STATUZ/PARAMS）按序 →
  `findBy<键…>Readr(实参…)`；键为空则放弃吸收（退回叶子路径）。
- **STATUZ-IF 形态不认识**（既非 ok/notok 也非 error，如掺入键字段）→ `_match_readr_single` 返回 None，
  整段不半吸收，退回叶子路径，宁可不动也不出错。
- **写操作（UPDAT/WRITE/DELET）**：本步骤结构吸收只覆盖「读」（返回记录、record!=null 语义）；
  写操作仍由 `_t_call` 叶子级 `obj=repo.save(obj)` 处理（无 record 返回，不适用 finder+null 模型）。

### 已知限制（未在本步骤处理，留待后续）
- **散点 `MOVE SPACES TO pfx-PARAMS`**（未被整段吸收的孤立场景）：仍渲染 `new XParams()`，
  非决策 §0 期望的「所有字段=''」——因拷贝簿字段表不在盘上、静态无法逐字段枚举（与决策 C 同类问题）。
  完整 READR 吸收路径里该行被 strip，故 §5 验证用例不受影响。
- **`BeanUtils` import**：生成代码用 `org.springframework.beans.BeanUtils`，import 由装配/骨架层统一管理，
  本步骤未触及该层；若装配层未自动登记需补 import。
