## 0. 已确认的决定 / 待确认决策点

### 关于「正本缺位」的说明（重要）
- 用户下发的翻译标准正本 `io查询.md` **只覆盖 READR / BEGN（读）**，写操作（UPDAT/WRITE/WRITR/WRITS/REWRT/DELET）**没有标准正本**。
- 但 `knowledge/io_call_patterns.md:186` 的「更新（UPDAT）」样例已编码了写约定（实体复用 + setter + save 丢返回 + STATUZ→try/catch）。READR 的确定性实现当初即对齐该 KB，故本步骤的写约定**以该 KB 样例为蓝本**，开放点单列在「待确认」。
- 落地前若用户补发写操作正本到 `docs/翻译标准/`，以正本为准（规范正本优先于 knowledge）。

### 已确认（来自 knowledge UPDAT 样例 + 步骤10 同构结论）
- `xxx-PARAMS` = 该 PO（实体）；写操作里 CALL 前 `MOVE val TO xxx-<字段>` 是给**实体字段赋值（setter）**，**不是查询键**（与 READR 相反，判别依据 = FUNCTION 功能码）。
- `xxx-FUNCTION` / `xxx-FORMAT` / `xxx-STATUZ` 是调用约定，须吸收（步骤10 已对全前缀生效，写操作自动继承，不再生成 setter）。
- 写操作 `save()` 的**返回值丢弃**：`xxxRepository.save(xxx);`（不 `xxx = repo.save(xxx)`，COBOL 写回 PARAMS 不被后续读）。
- 写后 `IF xxx-STATUZ NOT = O-K (PERFORM 9999/580)` → `try { 写 } catch (Exception e) { 错误段(); }`（与决策 A 同构；写无 record!=null/MRNF-null 语义，STATUZ 只分 ok / 错误两态）。

### ✅ 已确认（2026-06-08 用户「认可，继续」，各点采纳建议项）
- **决策 W-1（实体来源：复用 vs new）= 按 setup 是否含 `MOVE SPACES/INITIALIZE xxx-PARAMS` 区分**：含则 `new XxxRecord()`（插入），不含则复用上文 `xxx`（更新，不发实体行）。落地于 `_is_init_params` + `_render_write_single` 的 `is_new` 分支。
- **决策 W-2（save 实体类型）= 写实体一律 `XxxRecord`**：复用沿用上文变量 `xxx`（类型由上游 READR 定为 `XxxRecord`），new 场景 `XxxRecord xxx = new XxxRecord()`；吸收路径内不再出 `new XxxParams()`。
- **决策 W-3（STATUZ）= 选项②**：凡 `IF pfx-STATUZ NOT=O-K [AND …]`（每项均为 STATUZ 比较且含 `NOT=O-K`）→ try/catch 调错误段，不细分 MRNF。落地于 `_write_statuz_form`（返回 `'error'`/`None`，形态不认识则整体放弃不半吸收）。
- **决策 W-4（DELET）= 选项①** `repo.delete(xxx)`（传实体，免键派生）。已改 `config/mappings/io_mappings.yaml` 的 `DELET` 模板为 `delete({entity})`（叶子兜底同步受益）；结构渲染由 `_is_op_delete` 判 save/delete 两类。
- **决策 W-5（落地架构）= 复用步骤10 整段骨架**：新增 `_match_write_single`/`_render_write_single`，挂 `_rewrite_begn_loops` Pass 2c（BEGN 单次→READR 单次→写单次依次尝试；READR 命中即 `continue`，未命中再试写）。理由：try/catch 跨语句结构改写，叶子 `_t_call` 做不到（同决策 A）。

## 1. 目标与范围
把**单条写 IO**（save 类：UPDAT/WRITE/WRITR/WRITS/REWRT；delete 类：DELET）的调用约定整段吸收，
消除写后 `IF STATUZ` 的 `getStatuz()` 泄漏与 `new XxxParams()` 类型错误，对齐 knowledge UPDAT 样例。

**只做**：单条写 IO 的整段吸收（rules 引擎 + 按需改 `io_mappings.yaml` 的 DELET 模板）。
**不做**：读路径（READR/BEGN 已完成）、批量写/循环内写、LLM 兜底链路、拷贝簿字段表相关的散点 `new XxxParams()`（仍是已知限制，仅吸收路径内消除）。

## 2. 问题诊断（当前 leaf 路径产物的违规）
以 knowledge §186 UPDAT 样例过当前管线：

| # | COBOL | 当前译文 | 应为 |
|---|---|---|---|
| 1 | `MOVE UPDAT TO TMLCLST-FUNCTION` | （已吸收，无 setter）✅ | 同 |
| 2 | `MOVE WSAA-NEW-VALUE TO TMLCLST-FIELD` | `tmlclst.setField(wsaaNewValue)` ✅ | 同 |
| 3 | `CALL 'TMLCLSTIO' USING TMLCLST-PARAMS` | `tmlclst = tmlclstRepository.save(tmlclst);`（保留赋值；若无前置 READR，`tmlclst` 误为 `new TmlclstParams()`） | `tmlclstRepository.save(tmlclst);`（丢返回，实体为 `TmlclstRecord`） |
| 4 | `IF TMLCLST-STATUZ NOT = O-K`（PERFORM 9999） | `if (!"O-K".equals(tmlclst.getStatuz())) { error9999(); }` —— **getStatuz 泄漏** | `try { …save… } catch (Exception e) { error9999(); }` |

根因同步骤10：写后 STATUZ 检查未被整段吸收，落到普通 condition 翻译 → `getStatuz()` 泄漏；save 返回赋值 + 实体类型 Params 均为叶子路径残留。

## 3. 设计方案（确定性规则，无 LLM；待决策拍板后细化）

### 3.1 识别 `_match_write_single`（镜像 `_match_readr_single`）
- 定位 CALL，且前序 setup 有 `MOVE <写功能码> TO pfx-FUNCTION`（写功能码集合 = `operations` 中模板含 `save`/`delete` 的：UPDAT/WRITE/WRITR/WRITS/REWRT/DELET）。
- setup 起点：CALL 向前收集连续涉及 pfx 的语句（复用 `_stmt_touches_pfx`）。
- 区分 save / delete 两类（决策 W-4 影响 delete 渲染）。
- 紧随 CALL 的 `IF pfx-STATUZ …`：单项 `NOT=O-K` 或双项（按决策 W-3）→ `mode=error`（try/catch）；无 IF → `mode=plain`。

### 3.2 渲染 `_render_write_single`（镜像 `_render_readr_single`）
- **实体行**（决策 W-1/W-2）：含 `MOVE SPACES/INITIALIZE pfx-PARAMS` → `XxxRecord xxx = new XxxRecord();`；否则复用上文 `xxx`（不发实体行）。
- **setter 行**：CALL 前 `MOVE val TO pfx-<字段>`（排除 FUNCTION/FORMAT/STATUZ/PARAMS）→ `xxx.setField(val);`（复用 `_assign`/重绑定）。
- **save/delete 行**（决策 W-3/W-4）：`xxxRepository.save(xxx);` 或 `xxxRepository.delete(xxx);`（丢返回）。
- **STATUZ error 形**：save/delete + 后续包进 `try { … } catch (Exception e) { <错误段>(); }`（PERFORM 580/9999 移入 catch，consume_to=段末，同决策 A）。

### 3.3 挂接（决策 W-5）
- `_rewrite_begn_loops` Pass 2：BEGN 单次 → READR 单次 → **写单次** 依次尝试，均未命中退回叶子 `_t_call`。
- `_t_call` 写分支：未被整段吸收的散点写 CALL 仍按 `operations` 直出（保留兜底）。

### 实现要点 / 与设计的细节差异（回填）
- **setter 经叶子占位入管线**：`_render_write_single` 中 setter（`MOVE val TO pfx-字段`）经 `_render_rebound` → `build_skeleton` 渲染为 `/*__LEAF_n__*/` 占位并注册到 `ctx.leaves`，`_render_rebound` 内的 `_tag_rebind` 给 Stmt 打上「pfx→实体变量」标注，待后续 `translate_leaf` 时才渲染成 `xxx.setField(...)`（与 READR 体内重绑定同机制）。故骨架串里看不到 setter 字面，单测以 `translate_leaf` 断言。
- **功能码与 COBOL 动词撞名**：`MOVE WRITE TO X-FUNCTION` 中 `WRITE` 是 COBOL 动词，被 segmenter 在 `WRITE` 处误切；真实 LIFE 系统写功能码用 5 字缩写（WRITR/WRITS/REWRT/UPDAT/DELET），均不撞名。`io_mappings.yaml` 仍保留 `WRITE: save` 条目（兜底），但该裸写形态本就罕见且属 segmenter 既有限制，不在步骤11范围。
