# 步骤31 叶子翻译统一入口设计

状态：已实现（2026-06-30）

## 1. 背景

步骤18-23 已把 MOVE、IF 条件、PERFORM 循环壳、CALL、算术/赋值、控制流叶子词逐步下沉到 `translator.leaf`。但入口仍有两处：

- `translator.rules.translate_leaf` 通过 `rules._dispatch_leaf` 分派；
- `asg.visitor.LeafJavaVisitor.visit_Leaf` 自己按 `translate_arith_assign`、`translate_control` 顺序尝试。

这会让后续继续收敛叶子动词时需要同时维护两份入口顺序。

## 2. 目标与非目标

目标：

1. 在 `translator.leaf` 提供统一入口 `translate_leaf_stmt(tokens, ctx)`。
2. `rules.translate_leaf` 与 ASG `visit_Leaf` 共用同一入口。
3. 保留 `rules._dispatch_leaf` 作为兼容 wrapper，避免历史测试和工具断裂。
4. 不改变未支持动词的兜底语义：返回 `([], False)`，由调用方生成原有 TODO 文本。

非目标：

- 不迁移 STRING、UNSTRING、INSPECT 等新动词。
- 不删除 `rules._dispatch_leaf`。
- 不改变 `SectionJavaVisitor`、legacy SECTION fallback 或 diff 工具语义。

## 3. 设计

新增统一入口：

```python
def translate_leaf_stmt(tokens: list[str], ctx: LeafCtx) -> tuple[list[str], bool]:
    if not tokens:
        return [], False
    verb = tokens[0].upper()
    try:
        if verb == "MOVE":
            return translate_move(tokens, ctx)
        lines, ok = translate_arith_assign(tokens, ctx)
        if ok:
            return lines, ok
        if verb == "CALL":
            return translate_call(tokens, ctx)
        return translate_control(tokens, ctx)
    except (ValueError, IndexError):
        return [], False
```

分派顺序保持当前已迁能力：

1. MOVE；
2. INITIALIZE/SET/ADD/SUBTRACT/MULTIPLY/DIVIDE/COMPUTE；
3. CALL；
4. GO/GOBACK/STOP/EXIT/CONTINUE/NEXT。

`rules._dispatch_leaf` 改为：

```python
def _dispatch_leaf(toks: list[str], ctx: Ctx) -> tuple[list[str], bool]:
    return translate_leaf_stmt(toks, ctx)
```

`LeafJavaVisitor.visit_Leaf` 改为调用同一入口；未命中时仍输出 `// TODO-LEAF: ...`。

## 4. 测试策略

新增 `TestUnifiedLeafEntry`：

- 验证 `translate_leaf_stmt` 对 MOVE、算术/赋值、CALL 与旧 `_dispatch_leaf` 输出一致；
- 验证控制流叶子词可通过统一入口命中；
- 验证未迁动词继续返回 `([], False)`；
- 验证 `rules.translate_leaf` 与 ASG `visit_Leaf` 对支持叶子的输出一致。
