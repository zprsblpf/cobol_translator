# 步骤37：逻辑文档生成器设计

> 对应：处理逻辑树（logic tree）生成，从 ASG 分析 COBOL 程序 → 结果表反向追溯全路径 → JSON + Markdown 双输出。

---

## 一、设计思想

### 1. 从翻译到文档化

本项目已有的三相架构（解析 → ASG → visitor 翻译）完成了"COBOL → Java"的翻译。但翻译结果的**可理解性和可比对性**不足。

逻辑文档生成器在翻译之外新增一条产出线：**对于每个 COBOL 程序，生成一份"处理逻辑树"文档——从入口出发，按调用链展开到结果表写入/返回/终止，记录每条路径上的条件分叉、IO 操作和词元序列。**

### 2. 双层词元（已定稿）

- **语义层（主）**：按意图分组的词元，用于 COBOL vs Java 同拓扑 diff 比对。
- **动词层（辅）**：逐字记录 COBOL 原始动词，用于人工审阅。

语义层词元表（版本 1.0）：

| 类别 | 词元 | 说明 |
|------|------|------|
| 赋值 | `ASSIGN` | MOVE src TO dst（含 figurative ZERO/SPACES） |
| 归零重置 | `RESET` | MOVE SPACES/ZEROS TO 结构体（初始化） |
| 功能码注册 | `FUNC_SET` | MOVE READR/BEGN/WRITR TO xxx-FUNCTION |
| 条件分支 | `BRANCH_IF` | IF ... ELSE ... END-IF |
| 多路分支 | `BRANCH_SWITCH` | EVALUATE ... WHEN ... END-EVALUATE |
| 调用过程 | `CALL_PROC` | PERFORM section-name |
| 内联循环 | `LOOP` | PERFORM ... UNTIL/VARYING/TIMES ... END-PERFORM |
| 跳转 | `JUMP` | GO TO paragraph |
| 返回 | `RETURN` | EXIT / GOBACK / STOP RUN |
| 子程序调用 | `CALL_PROG` | CALL 'subprogram'（非 IO 型子程序） |
| DB 单条读 | `DB_READR` | CALL 'XXXIO' + FUNCTION=READR → repo.findByKey |
| DB 游标开始 | `DB_BEGN` | CALL 'XXXIO' + FUNCTION=BEGN → repo.findBy...Begn |
| DB 游标下条 | `DB_NEXTR` | CALL 'XXXIO' + FUNCTION=NEXTR → 循环内下条 |
| DB 插入 | `DB_WRITR` | CALL 'XXXIO' + FUNCTION=WRITR → repo.save |
| DB 更新 | `DB_UPDAT` | CALL 'XXXIO' + FUNCTION=UPDAT → repo.update |
| DB 删除 | `DB_DELET` | CALL 'XXXIO' + FUNCTION=DELET → repo.delete |
| 计算 | `MATH` | ADD/SUBTRACT/MULTIPLY/DIVIDE/COMPUTE |
| 文本操作 | `TEXT_OP` | STRING / UNSTRING / INSPECT |
| 搜索 | `SEARCH` | SEARCH ALL / WHEN |
| 错误终止 | `FATAL` | PERFORM 5xx-ERROR（系统错误处理段） |
| 未解析 | `UNRESOLVED` | 无法识别/未映射的语句（fallback 标记） |

### 3. 路径定义

路径 = **从入口到某个结果叶子的一条调用链**。

结果叶子（result leaf）分四类：

| 类型 | 标识 | 示例 |
|------|------|------|
| 表写入 | `table_insert` | WRITR → TMRDPF, ZTRDPF |
| 表更新 | `table_update` | UPDAT → HREPTRD |
| 表删除 | `table_delete` | DELET → TMLCLST |
| 错误终止 | `abend` | FATAL-ERROR |
| 空返回 | `return` | EXIT / GOBACK |

路径**不是**条件分支的全排列——它只记录**调用链**（PERFORM 嵌套路径）。条件分支在 `edgeConditions` 中记录，供路径级别的条件摘要使用。

### 4. 交叉点

当多条路径共享同一个 SECTION 节点时，该节点是交叉点。在逻辑树上只画一次，在 `pathTags` 中列出所有经过它的路径 ID。

## 二、模块设计

### 新模块：`logician/`

```
logician/
├── __init__.py          ← 门面 generate(program) → LogicDoc
├── models.py            ← LogicDoc 数据模型 (dataclass)
├── builder.py           ← 从 ASG 构建 LogicGraph
├── tokens.py            ← 词元提取（语义层 + 动词层）
├── results.py           ← 结果叶子识别
└── output/
    ├── __init__.py
    ├── json_output.py   ← LogicDoc → JSON 序列化
    └── md_output.py     ← LogicDoc → Markdown 渲染
```

### 数据模型（`models.py`）

```python
@dataclass
class LogicDoc:
    program_id: str
    schema_version: str = "1.0"
    side: str = "COBOL"           # "COBOL" | "Java"
    entry: LogicNode              # 入口节点
    results: list[ResultLeaf]     # 结果叶子清单
    paths: list[LogicPath]        # 路径列表
    nodes: dict[str, LogicNode]   # 节点字典 {nodeId: ...}
    edges: list[LogicEdge]        # 边列表

@dataclass
class ResultLeaf:
    id: str                       # "R-TMRD"
    kind: str                     # "table_insert" | "table_update" | ...
    target: str                   # "TMRDPF"
    paths: list[str]              # 经过此结果的路径

@dataclass
class LogicPath:
    path_id: str                  # "PATH-01"
    result_id: str                # "R-TMRD"
    node_sequence: list[str]      # ["E0", "B3", ...]
    semantic_tokens: list[str]    # 路径级别的语义摘要
    verb_tokens: list[str]        # 路径级别的动词摘要
    edge_conditions: list[EdgeCondition]

@dataclass
class LogicNode:
    node_id: str                  # "B3"
    type: str                     # "entry" | "box" | "fork" | "result"
    label: str                    # "1500-READ-ELPO"
    section: str | None           # COBOL SECTION 名
    semantic_tokens: list[str]    # 本节点的语义词元
    verb_tokens: list[str]        # 本节点的动词词元
    io: list[IOCall]              # IO 调用列表
    path_tags: list[str]          # 途经此节点的路径

@dataclass
class LogicEdge:
    from_id: str
    to_id: str
    kind: str                     # "always" | "conditional"
    branch: str | None            # "yes" | "no"
    expr: str | None              # 条件表达式
    path_ids: list[str]
```

### 核心流程（`builder.py`）

```
generate(program) → LogicDoc:
  1. 解析 ASG (asg.builder.build)
  2. 构建调用链图 (analyzer.callgraph)
  3. 追踪入口到所有调用路径
  4. 识别结果叶子节点（表写入/更新/删除）
  5. 对每个结果，逆向追溯完整调用链路径
  6. 提取每个节点的语义/动词词元
  7. 合并交叉节点（多条路径共享节点）
  8. 返回 LogicDoc
```

### 词元提取（`tokens.py`）

每个 ASG 节点按以下规则提取词元：

| ASG 节点类型 | 语义词元 | 动词词元 |
|-------------|---------|---------|
| `MoveStmt` | ASSIGN / RESET / FUNC_SET | MOVE |
| `CallStmt` by io_resolve | DB_READR / DB_BEGN / DB_WRITR ... | CALL |
| `CallStmt` other | CALL_PROG | CALL |
| `IfStmt` | BRANCH_IF | IF |
| `GotoStmt` | JUMP | GO |
| `Paragraph` 含 EXIT | RETURN | EXIT |
| `PerformStmt` out-of-line | CALL_PROC | PERFORM |
| `PerformStmt` inline loop | LOOP | PERFORM |
| 兜底 | UNRESOLVED | （首动词） |

## 三、IO 调用解析

IO 映射遵循 `docs/翻译标准/io查询.md` 的语义：

- `READR` → 单条读，`repo.findByKeyReadr(key)`
- `BEGN` → 游标定位，`repo.findBy...Begn(key)` 返回 List
- `NEXTR` → 游标下条（循环内嵌）
- `WRITR` → 插入，`repo.save(entity)`
- `UPDAT` → 更新，`repo.update(entity)`
- `DELET` → 删除，`repo.delete(entity)`

STATUZ 语义（已确认）：

| STATUZ | 含义 | 判定 |
|--------|------|------|
| O-K | 查询到数据 | ok |
| ENDP | 最后一条/无数据 | not ok |
| MRNF | 无数据 | not ok |

## 四、88 条件名处理

遵循 `docs/翻译标准/变量定义.md` §三 的规范：

```cobol
01  WSAA-A65086.
    03 WSAA-VAL28-PRD    PIC X(04).
    88 IS-6PWB           VALUES '6PWB'.
    88 IS-WPDH           VALUES 'WPDH'.

IF IS-6PWB THEN MOVE 'Y' TO WSAA-FLAG.
```

88 条件名在逻辑文档中展开为等价条件表达式：
- `IS-6PWB` → `WSAA-VAL28-PRD = '6PWB'`
- 展开前提：能从 WORKING-STORAGE 树中解析到 holder 字段（`WSAA-VAL28-PRD`）和值列表

未解析的 88 条件名在 `expr` 中标记为 `88:IS-6PWB`。
