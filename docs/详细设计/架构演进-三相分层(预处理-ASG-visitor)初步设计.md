## 0. 背景与动机：为什么"冗长"，以及偷什么

用户对比 `proleap-cobol-parser`（ANTLR4 前端）后提问：为何本项目冗长复杂。澄清两点：

1. **ProLeap 不译码**（评估 §1 实测）：它只做 `COBOL 文本 → Java 对象模型(AST/ASG)` 的**前端**，
   文法是声明式 `.g4`（ANTLR 自动生成 parser），**无任何 codegen/transform**。它"简洁"是因为只做最易自动化的那半。
2. **本项目冗长在翻译，不在解析**：`rules.py`（1328 逻辑行）做的是 COBOL 习语 → 地道 Java 的**语义映射**
   （PERFORM THRU 区间、BEGN/NEXTR 游标→for-each、IO CALL→Repository、PIC→类型、REDEFINES、88、GO TO→状态机）——
   这部分 ProLeap 压根不做，接了它也省不掉。

**复杂度分两类**：
- **本质复杂度**＝语义映射，不可压缩，任何 parser 都不送你。
- **偶然复杂度**＝`rules.py` 在 **token 层做模式匹配**（`tokens[0].upper()=="CALL"` 这类嗅探）、解析与翻译耦合、
  无稳定带类型中间模型。**这一半，ANTLR4 的"解析→带类型树→visitor"思想能削。**

**本演进只偷 ANTLR4 的思想（三段式 + visitor），不偷它的全 COBOL85 文法与零容错**——
后者实测在本 shop 方言（中文 DBCS、`GO` 省 `TO`、`!!!!!!` 不一致停用）仅覆盖 ~10%（评估 §6.2-6.3、§7.3）。
自研、方言鲁棒的前端是必保核心资产。

## 1. 目标三相架构

```
┌─ 相1 预处理 Preprocessing ───────────────────────────────────┐
│  COPY 展开 · 格式归一(定宽/自由/续行) · !!!!!! 停用行清理 ·            │
│  中文 DBCS 安全的列区切分(序号区/注释区剥离)                          │
│  产出：规范化「干净行流」(带原始行号回溯)                              │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─ 相2 解析 → 自研轻量 ASG ─────────────────────────────────────┐
│  词法/切分 → 建「带类型节点树」→ 符号/引用解析                         │
│  节点类型：Program / Division / Section / Paragraph /                 │
│            IfStmt / EvaluateStmt / PerformStmt / CallStmt /           │
│            MoveStmt / StringStmt / GotoStmt / Leaf…                   │
│  引用已解析：PERFORM 目标→真实过程单元、数据名→DDE、过程注册表          │
│  产出：稳定中性 IR —— 相3 的**唯一输入**                              │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─ 相3 翻译 Visitor ───────────────────────────────────────────┐
│  visit(IfStmt) / visit(PerformStmt) / visit(CallStmt) … → 地道 Java   │
│  config 规范正本继续驱动(类型/命名/IO/块构造)；兜不住的节点落 TODO     │
└──────────────────────────────────────────────────────────────┘
```

**设计思路**：三相单向数据流、各相边界清晰、可独立测试。相2 把"全程序语义"一次性算清（注册表 + 引用解析），
相3 的规则从此**只访问节点**，不再到处临时建索引、不再啃 token 序列。

## 3. IR / ASG 节点模型草图（初步，详设再定字段）

- **结构节点**：`Program → Division → Section → Paragraph`（带行号回溯、注册表索引）。
- **语句节点**（带类型，取代裸 token 块）：
  控制流 `IfStmt(cond, then, else)` / `EvaluateStmt(subject, whens)` / `PerformStmt(target, thru, until)` / `GotoStmt(target)`；
  叶子 `MoveStmt` / `SetStmt` / 算术 `ComputeStmt…` / `CallStmt(name, using)` / `StringStmt(sources, into)` / `Leaf(raw)`。
- **引用解析产物**：PERFORM 目标解析到真实过程单元（THRU 区间在此一次算清，替代步骤12/13 临时逻辑）；
  数据名解析到数据定义项(DDE)，供相3 判型/前缀。
- **兜底**：相2 无法归类的语句 → `Leaf(raw)` 节点，相3 visit 时落 `// TODO`（保守，不臆测，行为同今天）。

> 节点是否用 dataclass、visitor 用双分派还是 `visit_<type>` 命名约定、注册表数据结构——**详设步骤再定**。

## 4. 借鉴 / 不借鉴边界（出处：评估 §7）

**借（落进自研 Python 前端）**：① 解析→中性模型→翻译三段解耦；② visitor 化的翻译层；
③ 预处理独立成相；④ 全局过程注册表思想（proc_order 已印证方向）。

**不借（本 shop 崩点，§7.3）**：① 全 COBOL85 严格文法；② 零容错（缺拷贝簿硬抛、悬空 END-IF 致命中断、
按字符截 72 列致中文错位）。本项目"按需手搓 + 方言鲁棒"是反向优势，必保。

## 5. 增量绞杀式迁移分期

原则：**旧管线不停、随时可回退、现有 43 单测当回归闸**，每步对比产物。

1. **抽相1**：预处理独立成相，旧 segmenter 暂照吃其输出 → 零行为变化（纯重排，测试不变）。
2. **旁路建相2 ASG**：与旧 parser 并存，先只供新写的单点 visitor 消费，不影响旧路径。
3. **rules 逐类迁 visitor**：按动词一类一类迁（MOVE → IF → PERFORM/THRU → CALL → STRING …），
   每迁一类，全程序渲染产物逐字符比对旧路径（允许"更正确"的差异，需人工确认）。
4. **下线旧 token 路径**：全部迁完后删除旧 segmenter→rules token 通道，相2 ASG 成唯一输入。

每个分期点都是一个独立的「设计先行 → 认可 → 落地 → 回填」详设步骤。

## 6. 收益与代价（决策依据）

| | 内容 |
|---|---|
| **收益** | rules 从"啃 token 序列"→"访问带类型节点"，ad-hoc 形态识别（BEGN/NEXTR/READR/单写）清爽化；引用预解析后 PERFORM/数据名不再到处临时建索引；前端/翻译解耦更抗变化；rules.py 体量告警(1328行)随拆分自然缓解 |
| **代价** | 相2 的 ASG 节点模型 + 引用解析是**新增工作量**（最重）；绞杀期两套并存有维护/比对成本；迁移周期长 |
| **风险控制** | 绞杀式＝旧管线始终可用、43 测试当闸、逐类迁移可回退；非大爆炸重写 |
