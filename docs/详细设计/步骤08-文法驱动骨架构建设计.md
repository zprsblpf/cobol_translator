## 1. 方法论：为什么是「骨架 + 填充」而不是「逐行翻译」（已确认的决定）

把 SECTION 翻成 Java，不能一行 COBOL 对一行 Java。原因是 **COBOL 控制流不是一棵干净的树，
而是两层叠加**，必须先按文法切出**嵌套块骨架**，再往叶子里填固定样式：

| 层 | 内容 | 难度 | 已有承载 |
|---|------|------|---------|
| **段内** intra-paragraph | `IF/ELSE/END-IF`、`EVALUATE/WHEN`、`PERFORM…END-PERFORM`(内联)、算术/IO 上挂的 `ON SIZE ERROR/AT END/INVALID KEY` 隐藏分支 | 可递归切块成树 | `segmenter._Parser`（递归下降，已存在） |
| **段间** inter-paragraph | `GO TO`(任意跳转)、`PERFORM para THRU`(子程序调用)、顺序贯穿(fall-through) | **任意有向图，未必可归约** | `rules.build_section`（回跳→状态机降级，已存在） |

**核心难点（坐实）**：
1. **句点作用域二义**——老式 COBOL 不写 `END-IF`，一个句点可同时关掉多层嵌套 IF；块边界靠句点消解。
2. **标号三重身份**——paragraph 名同时是 GO TO 跳转目标、PERFORM 子程序入口、顺序贯穿的下一段，切块时要判角色。
3. **不可归约控制流**——GO TO 互相缠绕时整不成纯嵌套循环（编译原理可归约性问题），只能降级
   「程序计数器 + `while(true) switch(__pc)`」状态机模拟原 CFG。
4. **隐式副作用**——`PERFORM VARYING` 先测/后测、内联异常子句是挂在语句上的分支，不是一行对一行。

**关键纠偏（用户指示）**：解决这些**不允许在 `segmenter.py` 塞启发式魔数**（如 `lead<=4`）。
要与 WSAA 翻译同范式——**先把「切分文法」定义成 config 正本，再按正本写切分逻辑**。
即文法（列模型、块构造、词法字典、控制流降级策略）从代码上提为 `config/segmentation_spec.yaml`（正本），
`segmenter.py`/`cobol_columns.py` 经 `segmentation_loader` 取规则、照着切。

**两阶段定义**：
```
阶段A 骨架构建（文法驱动，输出嵌套块树）
  SECTION → split_paragraphs 切 paragraph（列模型驱动，修 Bug 1）
          → segment 递归切结构块 IF/EVALUATE/PERFORM/异常子句（块文法驱动）
          → build_section 段间控制流：回跳 GO TO 判可归约 → 结构化 / 状态机降级
阶段B 叶子填充（已有 rules 引擎）
  块树叶子(单语句 MOVE/ADD/SET…) 按固定样式翻译；赋值左值必为字段（修 Bug 2）
```

**不做**（出范围）：IF 复合条件/STRING/COMPUTE 等**新语句**翻译样式；LLM/RAG；graph；WSAA 生成；
GO TO DEPENDING ON 的计算跳转（留 TODO）。

## 2. 根因分析（探查结论，已坐实）

### Bug 1：paragraph 标号识别恒失败（切分入口坏 + 文法散落）
- `cobol_columns.clean_line` 取 `raw[6:72]`，**保留第 7 列（指示列）空格**：
  `       571-PARA.` → `' 571-PARA.'`（恒带 1 前导空格）。
- `segmenter.split_paragraphs:220` 用 `raw[:1] not in (" ","\t")` 判「Area A 标号」，净化行首恒为空格
  → 判定**永远为假** → 标号全落进语句体 → paragraph 整体塌成 1 段。
- 后果连锁：`build_section` 只拿到 1 个合成段 → `// paragraph` 注释 0 个、状态机 case 0 个、
  段内多段落结构整体丢失 → 大量本该是标号/EXIT 的行被当 leaf → `// TODO 叶子待译`（354 条）。
- **更深层**：列区划分、动词集、终止符集、标号正则全部**硬编码散落**在 `segmenter.py` 与
  `cobol_columns.py`，这正是要正本化的根因（不是补一个判断就完）。

实测「正确按列模型切分」后量化：具名标号 **324**；多段落 SECTION 129/133；
触发状态机降级（含回跳 GO TO）**27 段**；余 106 段走扁平拼接 + `// paragraph` 注释。

### Bug 2：赋值左值退化为字面量（翻译规范缺一条）
- `rules.py:192 _assign` 非结构体目标走 `_operand(dst)`；`_operand:177-178` 对「裸词 + 不在
  `field_type_map`」回退为字符串字面量——**对 RHS 操作数正确，对赋值左值错误**
  （MOVE/SET 目标语义上必为字段，绝不可能是字面量）。
- 翻译规范正本 `wsaa_translation_spec.yaml` **缺「赋值左值必为字段」这条**，故代码无依据兜底。
- 现象：`"WSAA-A65086" = "";`（417 处）。

## 3. 设计方案

### 3.1 两个 config 规范正本（按「物理切分」与「骨架文法」分文件）

config 是骨架的规范，本步把它落成**两个正本**，定位同 `wsaa_translation_spec.yaml`（抽象、可扩展、详尽）：
按职责分文件（守规则17）——物理列/词法切分归一个，COBOL→Java 的**骨架文法**单独一个。

#### (a) `config/segmentation_spec.yaml` —— 物理切分规范（列模型 + 词法 + paragraph）

```yaml
# ── 列模型（COBOL 固定格式列区，clean_line/Area 判定依据；去 6/72 魔数）──
column_model:
  indicator_col: 7              # 指示列：* / 注释、D 调试、- 续行、空 普通代码
  area_a: { start: 8,  end: 11 }    # SECTION/paragraph 名、01/77 级
  area_b: { start: 12, end: 72 }    # 过程语句、子级数据项
  clean_slice: { start: 7, end: 72 }  # clean_line 取此段（保留指示列）

# ── 词法字典（驱动切分的关键字集合，从 segmenter 上提）──
lexicon:
  verbs: [MOVE, COMPUTE, ADD, SUBTRACT, MULTIPLY, DIVIDE, PERFORM, GO, IF, EVALUATE,
          CALL, SET, INITIALIZE, STRING, UNSTRING, ACCEPT, DISPLAY, OPEN, CLOSE,
          READ, WRITE, REWRITE, DELETE, START, CONTINUE, NEXT, EXIT, STOP, GOBACK,
          RETURN, SEARCH, INSPECT]
  scope_terminators: [END-IF, END-PERFORM, END-EVALUATE, ELSE, WHEN, THEN]
  perform_keywords: [VARYING, UNTIL, WITH, TEST, BEFORE, AFTER, TIMES, THRU, THROUGH, FROM, BY]

# ── paragraph 模型（标号识别，修 Bug 1 的依据）──
paragraph_model:
  area: area_a                    # 标号必在 Area A；据列模型与 Area B 语句区分
  pattern: '^([A-Za-z0-9][A-Za-z0-9-]*)\.\s*$'
  exclude: [verbs, scope_terminators]   # EXIT./CONTINUE./ELSE. 不是标号
```

#### (b) `config/skeleton_spec.yaml` —— 骨架文法规范（COBOL 语法的格式化正本）

把 COBOL 过程语法**格式化**成构造表。沿用 `wsaa_translation_spec` 详尽风格：每条块构造声明
`cobol/meaning/header_until/regions/closed_by/java_skeleton/sample`。**这是项目核心资产**——
补 COBOL 语法即在此加构造、再改实现跟上。下为细化草案（落地即写成此文件）：

```yaml
# 字段释义（适用 block_grammar 每条）：
#   cobol         —— 该构造的 COBOL 语法形态
#   meaning       —— 语义 + 切分要点（句点作用域、嵌套）
#   trigger       —— 触发该构造的首关键字（_Parser 据此分派）
#   header_until  —— 头部（条件/主语/循环头）收集到何处停（token 类别集）
#   regions       —— 子区名 → 该子区 parse_statements 的 stop 集（递归切块边界）
#   closed_by     —— explicit: 显式终止符；implicit: 句点(period)/none
#   java_skeleton —— 对应 Java 骨架样式 id（实现侧据此渲染）
#   sample        —— cobol → java 对照

block_grammar:

  - id: if
    cobol: "IF cond [THEN] stmts... [ELSE stmts...] END-IF / ."
    meaning: |
      二分支。条件后可省略 THEN。then 区到 ELSE 或 END-IF 止；有 ELSE 则 else 区到 END-IF 止。
      旧式无 END-IF 时由句点闭合，且**一个句点可同时关掉多层嵌套 IF**（句点作用域）。
    trigger: IF
    header_until: [THEN, verb, period, scope_terminator]   # 条件 token 收集终点
    regions:
      then: [ELSE, END-IF]      # then 分支 stop 集（叠加外层 stop）
      else: [END-IF]            # else 分支 stop 集
    closed_by: { explicit: END-IF, implicit: period }
    java_skeleton: if_else
    sample:
      - cobol: "IF WSAA-FOUND = 'Y' MOVE 1 TO WSAA-N ELSE MOVE 0 TO WSAA-N END-IF"
        java:  "if (wsaaFound.equals(\"Y\")) { wsaaN = 1; } else { wsaaN = 0; }"

  - id: evaluate
    cobol: "EVALUATE subject WHEN v1 stmts WHEN v2 stmts [WHEN OTHER stmts] END-EVALUATE"
    meaning: |
      多分支（含 EVALUATE TRUE 形态）。先收主语，再逐个 WHEN：每个 WHEN 收条件值、其后语句为该分支体。
      WHEN OTHER 对应 default。各分支体到下一个 WHEN 或 END-EVALUATE 止。
    trigger: EVALUATE
    header_until: [WHEN, END-EVALUATE, period]   # 主语收集终点
    regions:
      whens: [WHEN, END-EVALUATE]                # 每个 WHEN 体的 stop 集（重复区）
    closed_by: { explicit: END-EVALUATE, implicit: period }
    java_skeleton: switch
    sample:
      - cobol: "EVALUATE WSAA-CODE WHEN 'A' PERFORM 100-A WHEN OTHER CONTINUE END-EVALUATE"
        java:  "switch (wsaaCode) { case \"A\": p100A(); break; default: break; }"

  - id: perform
    cobol: "PERFORM [para [THRU para2]] | PERFORM [VARYING..|UNTIL..|n TIMES] stmts END-PERFORM"
    meaning: |
      两形态：① out-of-line —— 头部为过程名(可带 THRU 范围)、无内联体 → 子程序调用；
      ② inline —— 头部为循环子句(UNTIL/VARYING/TIMES)、其后语句为循环体到 END-PERFORM 止 → 循环。
      头部遇到首个动词即认为内联体开始；遇句点/终止符/END-PERFORM 则无内联体。
    trigger: PERFORM
    header_until: [verb, period, scope_terminator, END-PERFORM]   # 循环头/过程名收集终点
    regions:
      body: [END-PERFORM]        # 内联循环体 stop 集
    closed_by: { explicit: END-PERFORM, implicit: none }
    java_skeleton: loop_or_call  # 有 UNTIL/VARYING/TIMES → while/for；裸过程名 → method-call
    sample:
      - cobol: "PERFORM 2000-INIT THRU 2000-EXIT"
        java:  "p2000Init();   // PERFORM 2000-INIT THRU 2000-EXIT"
      - cobol: "PERFORM UNTIL WSAA-EOF = 'Y' READ ELPO-FILE END-PERFORM"
        java:  "while (!wsaaEof.equals(\"Y\")) { readElpoFile(); }"

# 字段释义（control_flow）：段间 paragraph 级 CFG 的降级策略（build_section 实现侧据此选路）
control_flow:
  back_edge_to_state_machine: true   # 存在回跳 GO TO(目标段索引≤当前)→ FLOW:while(true)+switch(__pc)
  fall_through: continue_next        # 段尾无 transfer(GO TO/EXIT/GOBACK) → 顺序流入下一段
  flat_when_no_back_edge: true       # 无回跳 → 扁平拼接各段骨架 + `// paragraph <名>` 注释
  idioms:                            # 可归约惯用式，优先于状态机降级（已有改写）
    - id: begn_self_loop
      cobol: "段内 CALL…BEGN…IF…GO TO 自身(NEXTR 取下一条)"
      meaning: "BEGN+NEXTR 自跳读取循环 → 改写为 List + for-each，吸收 setFunction/跳出/NEXTR"
      java_skeleton: foreach
```

> 上为细化后草案；落地写入 `config/skeleton_spec.yaml` 时按真实 `_Parser` 行为微调 sample 文案。

### 3.2 新增 `config/grammar_loader.py`（COBOL 文法访问层，统管两正本）

**为何独立成文件**：`spec_loader.py` 已 125 行且专责「翻译规范」访问；COBOL 文法（物理切分 +
骨架文法）是另一职责，按规则 17 单列，与 `spec_loader` 并列、互不耦合。引擎经此层取规范，**不直读 yaml**。
本层同时加载 `segmentation_spec.yaml` 与 `skeleton_spec.yaml` 两正本并对外提供查询。

| 函数 | 来源 | 返回 | 职责 |
|------|------|------|------|
| `column_model()` | segmentation | dict | indicator_col / area_a / area_b / clean_slice |
| `verbs()` | segmentation | frozenset | 语句起始动词集 |
| `scope_terminators()` | segmentation | frozenset | 块作用域关键字 |
| `perform_keywords()` | segmentation | frozenset | PERFORM 内联循环关键字 |
| `paragraph_label(clean_line)` | segmentation | str\|None | **Bug 1 核心**：按列模型判 Area A 标号，是→返回大写名 |
| `block_grammar()` | skeleton | list[dict] | 段内块构造表（供 `_Parser` 实现对照与取参） |
| `back_edge_state_machine()` | skeleton | bool | 控制流降级开关（供 build_section 判定） |

`paragraph_label` 实现思路（**列模型驱动，非魔数**）：
- 设 `b0 = area_b.start - indicator_col`（=5，语句区在净化行的起始索引）。
- `lead = len(s) - len(s.lstrip())`（净化行前导空格数，含指示列那 1 格）。
- 判为标号 ⟺ `lead < b0`（名字落在 Area B 之前 = Area A 区）且 `pattern` 命中
  且 名称 ∉ `verbs∪scope_terminators`，是则返回大写名、否则 None。
- 「落在 Area B(lead≥5)」→ 语句。完全由 `column_model` 推导，兼容真实管线(lead 1–4)与理想化输入(lead=0)。

### 3.3 `segmenter.py` / `cobol_columns.py`：照文法切分（含 Bug 1 修复）
- `segmenter` 模块级 `VERBS/_TERMINATORS/_PERFORM_KW/_LABEL_RE` 改为**初始化时从 `grammar_loader`
  一次性载入**（保留模块级缓存变量，不每行 IO；引擎逻辑/递归结构不变）。
- `split_paragraphs:220` 的 Area A 判定改调 `grammar_loader.paragraph_label(clean_line)`（返回名或 None），
  **Bug 1 自然消除**——不再是 `raw[:1]` 魔数，而是列模型驱动；标号正则/排除集亦移入正本。
- `cobol_columns.clean_line:45` 的 `raw[6:72]` 改读 `column_model.clean_slice`（消除魔数 6/72）。
- `_Parser`（块文法递归切块）**是 `skeleton_spec.block_grammar` 的具体实现**：它对每个块构造
  （if/evaluate/perform）的「头部边界 / 子区 / 闭合」按规范递归切块。本步把其依赖的关键字集合、
  头部边界、子区 stop 集**全部改为从 `block_grammar()` 取参**，使代码与规范一一对应、不再硬编码。
  规范是正本、`_Parser` 服从它；以后规范加构造，再在 `_Parser` 补对应分支跟上（config 先行、实现跟随）。
- `build_section`（rules.py）**是 `skeleton_spec.control_flow` 的具体实现**：拿到正确 `paras` 后按
  策略走 `// paragraph`(106 段)/状态机(27 段)；`has_back_edge` 开关读 `back_edge_state_machine()`。

### 3.4 Bug 2：翻译规范补一条 + `rules.py` 照规范兜底
- `wsaa_translation_spec.yaml` 新增 `procedure_semantics` 节，一条规则 `assignment_target`：
  「MOVE/SET/COMPUTE 等赋值目标语义上必为字段，**绝不退化为字符串字面量**；未登记字段→仍按命名规则
  渲染为 camelCase Java 标识符。」（与数据节并列，作为过程语义正本的起点。）
- `rules.py` 新增 `_lvalue(dst, ctx)`：复用 `_operand` 的下标/引用修改分支，
  **裸词去掉「→字面量」回退**改 `_field_base(dst)`（camelCase）。
  **所有赋值目标点**改用 `_lvalue`：`_assign`（MOVE/SET/INITIALIZE）+ `_t_add/_t_subtract/_t_multiply/
  _t_divide` 的 GIVING/in-place 目标 + `_t_compute` 的 dst（照规范「MOVE/SET/COMPUTE 等」全覆盖）。
- 效果：`"WSAA-A65086" = "";` → `wsaaA65086 = "";`；`_operand` 的 RHS 行为**不变**（O-K / 未登记字段 RHS 仍为字面量，属数据登记缺口，非本步）。
