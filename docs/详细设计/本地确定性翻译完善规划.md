# 本地确定性翻译完善规划

日期：2026-07-01

## 目标

把当前 COBOL → Java 翻译器收敛为一条可持续扩展的本地确定性翻译主线：

- 用户补充翻译规范后，项目能按规范逐步固化为本地规则。
- 默认产品路径不依赖 LLM、RAG、MongoDB 或外部模型服务。
- 任何无法确定翻译的 COBOL 构造，都以结构化 unsupported 形式显式返回，而不是散落自由文本 TODO。
- 每次新增规范都有规则编号、实现入口、测试用例和覆盖率变化。

## 当前判断

当前架构方向正确，已经形成三相结构：

- 相1 `preprocess/`：固定列、注释/停用行、方言归一。
- 相2 `asg/`：从 token 语句提升为类型节点。
- 相3 `translator/`：`SectionJavaVisitor` + `translator.leaf` + `translator.skel` 确定性渲染。

当前主要缺口不是架构方向，而是产品化闭环：

- no-LLM 单文件路径 `execute()` 尚未实际调用入口 SECTION，且未生成完整 WSAA 容器。
- graph 多文件路径与 no-LLM 单文件路径并存，产品正本不够明确。
- `config/specs` 是规范访问层，不是通用规则执行器；新增语句仍需 Python 实现和测试。
- unsupported/TODO 文案不统一，难以统计规范缺口。
- 验证脚本能证明回归通过，但不能证明真实 COBOL 覆盖完整。

## 产品主线决策

后续以“本地确定性多文件输出”为产品正本。

推荐目标命令：

```bash
python main.py input.cob --deterministic --output out
```

目标行为：

- 只走本地确定性解析和翻译。
- 不调用 LLM。
- 支持的 COBOL 构造直接输出 Java。
- 不支持的构造输出结构化 `UNSUPPORTED[...]` 注释，并写入机器可读报告。
- 输出 Java 文件、`unsupported_report.json`、`coverage_report.md`、`review_checklist.md`。

LLM/RAG 路径保留为开发辅助，不作为默认产品验收标准。

## 阶段 1：产品主线收敛

### 目标

明确 deterministic 产品路径，修复当前 no-LLM 输出不完整的问题。

### 主要任务

1. 增加 CLI 模式：

```bash
python main.py input.cob --deterministic --output out
```

语义上等同“多文件、本地规则、无 LLM”。

2. graph 翻译节点支持 deterministic：

- `translate_section_node` 在 deterministic 模式下禁止调用 `_translate_leaves_llm`。
- 规则未命中的叶子直接生成 structured unsupported。

3. 修复单文件 no-LLM 路径：

- `translator/skeleton_gen/render_skeleton.py::_entry` 实际调用首个 SECTION。
- 如果继续保留单文件输出，需要补齐或内嵌 `Wsaa` 容器类；否则明确标注该路径仅用于片段调试。

4. ASG fallback 可观测：

- `body_context.translate_paragraphs_body` 发生 ASG → legacy fallback 时，记录到 review/unsupported 报告。

### 涉及文件

- `main.py`
- `graph/nodes.py`
- `graph/state.py`
- `translator/skeleton_gen/render_skeleton.py`
- `translator/skeleton_gen/body_context.py`
- `translator/assemble.py`

### 验收

```bash
python scripts/check.py
python main.py tests/fixtures/minimal.cob --deterministic --output /tmp/cobol-det-minimal
```

验收标准：

- 测试全绿。
- deterministic 模式不访问 LLM。
- minimal 产物入口可实际进入首个 SECTION。
- 无匿名 `// TODO: 等待 LLM 翻译` 残留。

## 阶段 2：结构化 unsupported 体系

### 目标

把当前散落的 TODO 统一成可统计、可追踪、可回归的 unsupported 事件。

### 设计

新增：

```text
translator/unsupported.py
```

建议接口：

```python
def unsupported_comment(rule_id: str, kind: str, raw: str, reason: str) -> str:
    ...

def unsupported_event(rule_id: str, kind: str, raw: str, reason: str, section: str = "") -> dict:
    ...
```

Java 注释格式：

```java
// UNSUPPORTED[LEAF.UNSTRING.001]: reason=rule_missing raw="UNSTRING ..."
```

JSON 事件格式：

```json
{
  "rule_id": "LEAF.UNSTRING.001",
  "kind": "leaf",
  "reason": "rule_missing",
  "section": "1000-MAIN",
  "raw": "UNSTRING ..."
}
```

### 首批接入点

- `asg/visitor.py::visit_Leaf`
- `asg/visitor.py::visit_CallStmt`
- `asg/visitor.py::visit_PerformStmt`
- `asg/visitor.py::visit_EvaluateStmt`
- `asg/section_visitor.py`
- `translator/skel/perform_call.py`
- `translator/skeleton_gen/body_context.py`
- `graph/nodes.py`

### 规则编号建议

- `LEAF.UNSTRING.001`
- `LEAF.INSPECT.001`
- `LEAF.SEARCH.001`
- `LEAF.STRING.OVERFLOW.001`
- `FLOW.GOTO.UNKNOWN.001`
- `FLOW.PERFORM.THRU.UNRESOLVED.001`
- `COND.88.UNSUPPORTED.001`
- `DATA.REDEFINES.ARRAY.001`
- `DATA.NAME_COLLISION.001`

### 验收

```bash
python -m unittest test_translation -v
python scripts/check.py
```

验收标准：

- 新增 unsupported 单测。
- 新代码不再新增匿名 TODO。
- 旧 TODO 可以分批迁移，但新增规则必须使用 `UNSUPPORTED[...]`。

## 阶段 3：规则覆盖扫描器

### 目标

### 新增命令


### 数据来源

- `parser.cobol_parser.parse`
- `asg.build_asg`
- `translator.leaf.translate_leaf_stmt`
- `SectionJavaVisitor`

### 验收

验收标准：

- 未支持语句必须带 rule_id 和 reason。
- 报告可用于指导下一条规范实现。

## 阶段 4：规范资产化

### 目标

把 docs 中的翻译规范变成可追踪规则资产。

### 新增目录

```text
config/rules/
  leaf_unstring.yaml
  leaf_inspect.yaml
  leaf_search.yaml
  leaf_string.yaml
  flow.yaml
  data.yaml
```

### 规则模板

```yaml
id: LEAF.UNSTRING.001
verb: UNSTRING
status: planned
title: "UNSTRING DELIMITED BY INTO basic form"
description: |
  ...
supported_when:
  - "单一 source"
  - "DELIMITED BY 字面量或 SPACE"
  - "INTO 一个或多个目标字段"
unsupported_when:
  - "WITH POINTER"
  - "TALLYING IN"
  - "ON OVERFLOW / NOT ON OVERFLOW"
examples:
  - cobol: "UNSTRING WSAA-A DELIMITED BY SPACE INTO WSAA-B WSAA-C"
    java: "..."
tests:
  - "test_translation.TestLeafUnstringExtract.test_basic_space"
implementation:
  module: "translator/leaf/unstring.py"
  dispatcher: "translator/leaf/__init__.py::translate_leaf_stmt"
```

### 原则

- YAML 先作为规则登记和覆盖资产，不一开始做通用模板解释器。
- Python 执行器仍然负责复杂翻译逻辑。
- 每个 rule_id 必须能关联文档、实现和测试。

### 验收

- 至少迁入 `UNSTRING`、`INSPECT`、`SEARCH` 三个 planned 规则文件。
- 覆盖扫描器能读取 `config/rules` 并标注 planned/implemented。

## 阶段 5：按覆盖缺口补叶子规则

### 优先级

1. `UNSTRING`
2. `INSPECT`
3. `SEARCH`
4. 复杂 `STRING WITH POINTER / ON OVERFLOW`
5. `EVALUATE TRUE`
6. 88 条件名
7. `GO TO DEPENDING ON`
8. OF/IN 字段限定消歧
9. REDEFINES/数组视图增强

### 每条规则的开发步骤

1. 在 `config/rules/*.yaml` 登记 rule_id。
2. 在 `docs/翻译标准/` 补充语义说明。
3. 新增或更新 `translator/leaf/<verb>.py`。
4. 接入 `translate_leaf_stmt`。
5. 补充单元测试：

```text
TestLeaf<Verb>Extract
TestUnifiedLeafEntry
TestAsg<Verb>Visitor（如需要）
```

6. 跑覆盖扫描器，确认 unsupported 数下降。
7. 更新操作记录。

### 验收

```bash
python -m unittest test_translation.TestLeafUnstringExtract -v
python -m unittest test_translation.TestUnifiedLeafEntry -v
python scripts/check.py
```

## 阶段 6：验证与质量门禁增强

### 目标

让后续开发可以快速验证，也可以完整验收。

### `scripts/check.py` 建议新增 suite

```bash
python scripts/check.py --suite quick
python scripts/check.py --suite leaf
python scripts/check.py --suite asg
python scripts/check.py --suite deterministic
python scripts/check.py --suite all
```

### 建议门禁

- 新增代码不能引入匿名 TODO。
- unsupported 必须有 rule_id。
- deterministic 模式不能调用 LLM。
- ASG fallback 必须进入报告。
- 真实样本 coverage 不能比基线恶化，除非更新基线并说明原因。

## 阶段 7：文档同步

每完成一个阶段，至少更新：

- `docs/操作记录/步骤XX-*.md`
- `docs/架构索引/项目总览.md`
- 对应 `docs/翻译标准/*.md`
- 对应 `config/rules/*.yaml`

文档职责分工：

- `docs/翻译标准/`：业务语义和人工可读规范。
- `config/rules/`：机器可追踪规则资产。
- `docs/详细设计/`：实现方案。
- `docs/操作记录/`：实际完成内容、验收命令、边界。
- `docs/架构索引/项目总览.md`：当前状态索引。

## 最终完成标准

项目达到以下状态时，可认为本地确定性翻译主线成型：

- 默认产品命令不依赖 LLM。
- 所有不支持构造都有结构化 rule_id。
- 覆盖扫描器能驱动规范补齐。
- 新增规范有固定开发流程：文档 → rule_id → 实现 → 测试 → coverage 下降。
- graph/LLM 路径仅作为开发辅助存在，不影响本地主线验收。
