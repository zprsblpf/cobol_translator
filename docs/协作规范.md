# 协作规范（COBOL→Java 翻译项目）

> 用途：本文件固化「我（Claude）与用户」的协作约定。
> 用法：**当用户让我「读协作规范 / 按规范来 / 读 docs/协作规范.md」时，我直接遵守以下全部条款，无需再逐条沟通确认。**
> 维护：约定有更新时追加/修订本文件，并在文末「变更记录」登记。

---

## 一、核心思想：设计先行（最高优先级）

1. **任何代码动作之前，先给详细设计，用户认可后才写代码。** 顺序固定：
   `给详细设计 → 用户认可 → 写代码 → 回填实现细节到设计文档`。
2. 详细设计必须讲清楚：
   - 创建的**每个类**是干什么的；
   - **每个方法**具体实现了什么功能；
   - 背后的**设计思路**。
3. 未经认可，**不擅自开始写代码、不跑有副作用的命令**。只读探查（grep/read/结构扫描）可以先做。
4. 用户主导节奏：用户会**分层逐步下发**规则（顶层设计思想 → 拆分规则 → 每块翻译规则，细到每个小功能）。
   我在每一层都先产出设计、等认可、再落地，不跨层抢跑、不超出当前步骤范围。

## 二、流程纪律

5. **不越界**：用户说"只做 X"时，严格只做 X。不提前做后续步骤的事（如本该只拆解时不要去翻译）。
6. **作废即停**：用户说"作废/停"，立即停止当前所有动作，不狡辩、不续做。
7. **决策固化**：用户做出的选择（目录、命名、取舍等）记入对应设计文档的「已确认的决定」节，后续不再反复问。
8. **不臆测关键参数**：连接串、密码、主机端口等不猜测、不内置默认值；缺失时显式报错或留待用户提供。

## 三、文档与产物

9. **设计文档**：放 `docs/详细设计/`，每个步骤/功能块一份。状态生命周期：
   `🟡设计中 → 🔶待认可 → ✅已认可 → 🟢已实现`。
10. **操作文档**：每个重要步骤执行完，生成一份操作记录放 `docs/操作记录/`，供用户留存/复盘
    （含命令、产物路径、校验结果）。
11. **拆解/翻译产物**：放 `/home/zp/Documents/cob/拆解/{PROGRAM}/`（按用户指定）。
12. **脚本一律保留**：所有用到的脚本（含中间校验脚本）都作为**正式文件**保存在 `scripts/` 或 `db/` 下，
    **不创建用完即删的临时脚本**，后续会复用。

## 四、代码编写规范

13. 脚本/模块**参数化、可复用**（接受程序名、路径等参数，不硬编码单一程序）。
14. 每个文件顶部写**模块 docstring**：用途、对应设计文档、用法示例。
15. 每个类/函数写清**做什么 + 设计思路**（中文注释，与现有代码风格一致）。
16. 复用现有能力，不重复造轮子；确需另起炉灶时在设计文档里说明原因
    （如本项目 `parser/cobol_parser.py` 不识别 `!!!!!!`，故另写 `scripts/decompose.py`）。
17. 写完即**自检并向用户汇报**：跑校验脚本，逐项列出通过情况，失败如实报告。

## 五、COBOL 领域固定约定（已确认）

18. **注释/停用判定**：
    - 指示列（第 7 列）= `*` 或 `/` → **注释**，**保留**（含业务语义，对翻译有用）。
    - 第 1–6 列 = `!!!!!!` → **停用旧代码**，**丢弃**（含 `!!!!!!` 开头的 SECTION 头，不作为边界）。
19. **拆解命名规则**：
    - `PROGRAM-ID. xxx` → 类名 `xxx`。
    - `WORKING-STORAGE SECTION` → 参数定义块，命名 `{PROGRAM}WSAA`。
    - `COPY`：`REC`/`SKM` 结尾 ≈ 实体类字段定义；含 `VARCOM` ≈ 处理逻辑/方法；其余 = other（只分类，不展开）。
    - `PROCEDURE DIVISION USING ...` → 方法入口 + 入参。
    - PROCEDURE 内每个有效 `SECTION` → 一个方法体 / 一个切片文件。
20. **切片保真度**：去掉行号列（1–6）与行尾 `<变更标记>`，保留第 7–72 列代码区与 `*` 注释。

## 六、知识库（MongoDB）

21. 库 `cobol`，集合名 = cob 程序名（每程序一表），每个块一条文档。
22. 文档预留字段 `params_in / params_out / calls / logic / java_code`，按步骤逐步回填。
23. 写库用**幂等 upsert**（键 `{program, block_name}`），重跑不重复、不覆盖已回填字段。
24. Mongo 不可用时**先写文件**，`manifest.json` 含全部入库字段，就绪后用同脚本 `--import-only` 补入库。

## 七、六步流水线（总览）

1. **拆解 + 存库**（不翻译）← 已完成
2. 每块的处理逻辑 + 代码生成规则
3. 按块生成 Java 代码
4. 组装
5. 校验
6. 按原 cob 校验逻辑

> 约定：Java 生成后库内方法体可能变更，故库结构支持更新（预留字段 + upsert）。
